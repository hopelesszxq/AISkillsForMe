## 概述

PostgreSQL 19（预计 2026 年下半年发布）在运维、性能、复制三大维度做了重要升级。核心变化：REPACK 命令替代 VACUUM FULL/CLUSTER、分区表支持 MERGE/SPLIT、自动清理使用并行工作线程、默认 TOAST 压缩改为 lz4。

## 1. REPACK 命令（替代 VACUUM FULL 和 CLUSTER）

### 功能说明

新增 `REPACK` 命令，统一替代 `VACUUM FULL` 和 `CLUSTER`，在在线模式下重建表和索引，**无需长时间阻塞写入**。

```sql
-- 在线重建表（替代 VACUUM FULL）
REPACK TABLE orders;

-- 在线重建表并按指定索引排序（替代 CLUSTER）
REPACK TABLE orders USING idx_orders_created_at;

-- 只重建索引
REPACK TABLE orders INDEXES ONLY;

-- 查看所有需要 REPACK 的表
SELECT schemaname, tablename, n_dead_tup, n_mod_since_analyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 100000;
```

### 配置项

```conf
# postgresql.conf
# REPACK 使用的复制槽数量（默认 1）
max_repack_replication_slots = 2
```

### 与传统命令对比

| 特性 | VACUUM FULL | CLUSTER | REPACK |
|------|------------|---------|--------|
| 阻塞写入 | ✅ 长时间阻塞 | ✅ 长时间阻塞 | ❌ 在线重建 |
| 索引排序 | ❌ | ✅ | ✅ |
| 复制槽依赖 | ❌ | ❌ | ✅（用于 CDC 同步） |

> **注意**：REPACK 依赖逻辑复制槽，需要设置 `max_repack_replication_slots`。如果不需要变更数据捕获同步，可设为 0 禁用。

## 2. 分区合并与拆分

### 功能说明

`ALTER TABLE` 现支持直接**合并（MERGE）**和**拆分（SPLIT）**分区，无需手动建表和数据迁移。

```sql
-- 将两个分区合并
ALTER TABLE orders MERGE PARTITIONS p2026_q1, p2026_q2 INTO p2026_h1;

-- 拆分一个分区为多个（按指定边界）
ALTER TABLE orders SPLIT PARTITION p2026_q1 INTO
    (PARTITION p2026_jan VALUES FROM ('2026-01-01') TO ('2026-02-01'),
     PARTITION p2026_feb VALUES FROM ('2026-02-01') TO ('2026-03-01'),
     PARTITION p2026_mar VALUES FROM ('2026-03-01') TO ('2026-04-01'));
```

### 适用场景

- 时间分区滚动：季度分区 → 年度分区归档
- 热点分区拆分：某个分区数据量过大，拆分为更细粒度
- 分区策略调整：动态调整分区边界

## 3. 并行自动清理（Autovacuum Parallel Workers）

### 功能说明

Autovacuum 现在可以使用**并行工作线程**来加速清理大表。新增参数控制并行度。

```conf
# postgresql.conf
# autovacuum 可使用的最大并行工作线程数（默认 0 = 禁用）
autovacuum_max_parallel_workers = 2

# 按表级别覆盖（也可以在表存储参数中设置）
ALTER TABLE huge_table SET (autovacuum_parallel_workers = 4);
```

### 自动清理评分系统

新增**评分系统**来决定表的清理优先级，避免小表频繁清理而大表被忽略：

```conf
# postgresql.conf
# 启用评分系统（替代原有按死元组数量排序）
autovacuum_scoring_enabled = on
```

评分因素包括：死元组比例、表大小、上次清理时间、年龄等。

## 4. 默认 TOAST 压缩改为 lz4

```conf
# postgresql.conf
# 默认从 pglz 改为 lz4（PG19 默认值）
default_toast_compression = lz4
```

### 性能对比

| 特性 | pglz（旧默认） | lz4（PG19 默认） |
|------|---------------|------------------|
| 压缩速度 | ❌ 较慢 | ✅ 快 3-10 倍 |
| 解压速度 | ❌ 较慢 | ✅ 快 2-5 倍 |
| 压缩率 | ✅ 稍高 | ⚠️ 略低（可接受） |
| CPU 开销 | 高 | 低 |

> 对于已存在的数据，不会自动重新压缩。需要执行 `ALTER TABLE ... SET COMPRESSION lz4;` 并重写表才能生效。

## 5. oid8 — 64 位无符号整数类型

新增 `oid8` 数据类型，用于存储 64 位无符号整数，扩展了 `oid`（32 位）的能力。

```sql
-- 创建使用 oid8 的表
CREATE TABLE large_objects (
    id    oid8 PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    data  text
);

-- 系统视图中的 oid8 列
SELECT oid8_table_name, oid8_column_name
FROM pg_oid8_map;
```

## 6. 查询优化器增强

### Memoize 增强

```sql
-- EXPLAIN 现在显示 Memoize 缓存和查找统计
EXPLAIN (ANALYZE, MEMOIZE) SELECT ...;
-- 输出示例：
--   Memoize  (cost=... rows=...)
--     Cache hits: 9500  Cache misses: 500  Cache evictions: 0
```

### 半连接优化

```sql
-- PG19 改进了半连接（SEMI JOIN）的规划
-- EXISTS / IN 子查询生成更优的执行计划
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM payments p WHERE p.order_id = o.id);
```

### 聚合下推

部分聚合操作可以在 JOIN 之前执行，减少 JOIN 数据量：

```sql
-- PG19 可以将 COUNT/SUM 聚合提前到 JOIN 之前
SELECT d.name, COUNT(o.id)
FROM departments d
LEFT JOIN orders o ON o.dept_id = d.id
GROUP BY d.name;
```

### Radix Sort

```sql
-- PG19 使用基数排序（Radix Sort）加速特定场景的排序操作
-- 对整数、日期等类型，超过阈值自动使用 radix sort
-- EXPLAIN 输出会显示排序方法
EXPLAIN (ANALYZE) SELECT * FROM large_table ORDER BY id;
-- Sort Method: radix sort  Memory: 65536kB
```

### 外键约束检查性能提升

```sql
-- PG19 显著加速了外键约束检查（特别是批量插入/更新时）
-- 无需任何配置，自动受益
```

## 7. 在线启用/关闭数据校验和

```sql
-- PG19 支持在线切换 data checksums，无需停机
ALTER SYSTEM SET data_checksums = on;
-- 或
ALTER SYSTEM SET data_checksums = off;

-- 查看数据校验和状态
SELECT * FROM pg_data_checksums;
```

> **注意**：在线切换后，新写入的页面会按新设置处理，旧页面不会自动重写。

## 8. 逻辑复制增强

### 序列同步

```sql
-- 允许订阅者序列值自动与发布者同步
CREATE PUBLICATION pub1 FOR ALL TABLES WITH (PUBLISH_SEQUENCES);

-- 刷新序列到订阅端
ALTER SUBSCRIPTION sub1 REFRESH SEQUENCES;

-- 查看序列同步状态
SELECT * FROM pg_stat_subscription_sequences;
```

### 并发复制设置

```sql
-- 允许订阅指定多个并行复制线程
CREATE SUBSCRIPTION sub1 CONNECTION '...' PUBLICATION pub1
    WITH (parallel_sync_workers = 4);
```

## 9. 外键约束强制执行开关

```sql
-- ALTER TABLE 现在支持对任意约束启用/禁用强制执行
-- 之前仅支持外键约束
ALTER TABLE orders ALTER CONSTRAINT fk_user_id ENFORCED;
-- 或
ALTER TABLE orders ALTER CONSTRAINT fk_user_id NOT ENFORCED;
```

## 10. 其他实用改进

| 特性 | 说明 |
|------|------|
| `COPY TO` 分区表 | `COPY partitioned_table TO '/tmp/data.csv'` 现在无需通过 `SELECT` 子查询 |
| `json_array()` 返回空数组 | 无匹配行时返回 `[]` 而非 `NULL` |
| `jsonpath` 新增字符串方法 | `starts_with()`, `ends_with()`, `contains()`, `substring()` 等 |
| 可见性图标记 | 表扫描时自动标记全可见页面，减少 `VACUUM` 负担 |
| `ALTER TABLE ... SET COMPRESSION` | 仅用 `ACCESS EXCLUSIVE` 而非全表重写 |
| `oid8` 数据类型 | 64 位无符号整数，适合超大 ID 场景 |
| `standard_conforming_strings` 强制开启 | 兼容性提升，旧式转义不再支持 |
| `MD5` 密码认证警告 | 成功后发出弃用警告，推动迁移到 SCRAM-SHA-256 |
| `random(min, max)` | 新增 date/timestamp/timestamptz 版本的随机函数 |
| `bytea ↔ uuid` 互转 | 支持直接 CAST |

## 升级注意事项

1. **REPACK 依赖复制槽**：如使用 REPACK 功能，确保 `max_replication_slots` 和 `max_repack_replication_slots` 设置充足
2. **lz4 默认压缩**：升级后新写入的数据使用 lz4，旧数据需手动重写
3. **`standard_conforming_strings` 强制开启**：如果应用依赖旧式转义，需适配
4. **MD5 认证弃用**：建议迁移到 `password_encryption = scram-sha-256`
5. **锁结构大小变更**：锁配置需要翻倍以匹配旧版本的容量
6. **`get_relation_info_hook` 已移除**：如果使用了自定义 hook 扩展，需要迁移到 `build_simple_rel_hook`
7. **数据校验和在线切换**：切换后旧页面的校验和状态不变
8. **分区 MERGE/SPLIT**：操作期间会阻塞分区上的写入，请在维护窗口执行
