## 要点

MinIO Community Edition 已不再适合新生产部署。自 2025 年起，上游分发模式发生了实质性变化：

| 时间 | 变化 | 影响 |
|------|------|------|
| 2025-05 | 关键管理功能移出 CE 路径 | CE 的 auth/admin 能力降低 |
| 2025-10 | 社区 Docker 镜像和公共二进制包停止分发 | 必须从源码自行构建验证 |
| 2025-12 | 明确进入维护模式 | 无活跃 OSS 迭代预期 |
| 2026-02 | 仓库首次归档 | 只读状态，阻止正常协作 |
| 2026-04 | 仓库再次归档并保持锁定 | 上游长期冻结确认 |

> 代码许可证仍为 AGPLv3，但实践已从"开箱即用"变为"源码自行构建"——对大多数团队而言，这更像"提供源码"而非"活着的开源生态"。

## 风险分析

| 风险维度 | 说明 |
|----------|------|
| **补丁节奏风险** | 无稳定可信的社区二进制通道，每个 CVE 周期都变成你团队的构建+发布周期 |
| **验证负担** | 团队必须自行负责来源验证、可重复构建和回滚策略 |
| **生态漂移风险** | 依赖公共镜像的工具可能滞后或失效 |
| **人力成本** | 高级 SRE/安全人员的时间消耗在基础设施管道上，而非产品工作 |

## 替代方案对比

### SeaweedFS
- **适用场景**：海量小对象、普通硬件上的高效存储
- **优势**：Apache 2.0 许可、高小对象密度、生产就绪且比 Ceph 轻量
- **不适用**：需要完整 S3 语义兼容性

### Garage
- **适用场景**：轻量自托管集群、边缘节点、地理分布式部署
- **优势**：资源高效、支持 ARM/小节点环境、运维简单
- **不适用**：需要完整 S3 特性集

### RustFS
- **适用场景**：实验性迁移、接受 Alpha 阶段风险
- **优势**：部署和 UX 最接近 MinIO 风格、发展迅速
- **不适用**：监管/高可用生产系统（保持 Pilot 阶段）

### Ceph RGW
- **适用场景**：企业级大规模部署、严格 S3 兼容性要求
- **优势**：成熟的企业 S3 行为、广泛能力
- **不适用**：小型团队、低运维预算

## 选择决策

```
小团队 + 低运维预算 → Garage 或 SeaweedFS
大型企业 + 严格兼容 → Ceph RGW
MinIO 风格迁移实验 → RustFS（Pilot，保留回滚选项）
```

## 迁移计划（四阶段）

### Phase 1：盘点与风险评估
- 列出所有 Bucket、对象数量、增长率
- 按关键性和恢复目标分类负载
- 识别硬 S3 依赖（版本控制、对象锁定、策略行为）

### Phase 2：兼容性验证
- 搭建 1~2 个候选平台（SeaweedFS / Garage / Ceph RGW）
- 重放代表性读写负载
- 验证认证、生命周期规则、保留行为、SDK 边界情况

### Phase 3：Pilot 切换
- 迁移一个低爆炸半径的负载
- 尽可能使用双读验证
- 测量延迟、错误率和运维开销

### Phase 4：生产迁移
- 优先迁移面向用户的高优先级负载
- 保留回滚制品和保留窗口
- 退役 MinIO CE 前完成最终 Runbook

## 注意事项

1. **不要恐慌性关闭**：已有 MinIO CE 部署视为可控技术债务，规划迁移窗口
2. **尽早做迁移决策**：在自己可控的时间窗口内完成，而不是在下次强制变更前仓促应对
3. **社区共识**：2025~2026 年趋势是更少团队选择 MinIO CE 作为新部署，更多团队优先评估 Garage 和 SeaweedFS
4. **合规场景注意**：AGPLv3 许可是否满足企业合规要求，需与法务确认
5. **备份策略**：迁移期间保留 MinIO CE 数据和配置备份，确保可回滚
